## 目的

外部/協業の開発パートナーが、REDISH SFA の状況を最短で把握し、ローカル起動〜最初のPRまで到達するための入口。

## まず読む（30分）

- `PROJECT_CONTROL.md`（正本 / 進捗SSOT）
- `docs/📌 docs_INDEX.md`（資料の入口）
- `docs/01_企画書/SFA_Webアプリ化_企画書・要件定義書.md`（要件・スプレッドシート前提）
- `docs/04_ルール・ガイドライン/開発ルール.md`（規約）
- `docs/04_ルール・ガイドライン/デザインシステム.md`（UI）
- `docs/05_環境構築・デプロイ/setup_guide_supabase_vercel_github.md`（環境構築・Google OAuth）
- `docs/05_環境構築・デプロイ/deployment_supabase_vercel.md`（デプロイ）

## 受け取るべきアクセス/権限（依頼者側の準備）

### GitHub

- リポジトリ閲覧/Push権限（推奨: PR運用）
- `main` 保護ルール、レビュー担当者、マージ方法（squash等）

### Vercel

- プロジェクトへのメンバー招待（Deploy履歴/Logsが見られる権限）
- 環境変数の設定・更新権限（少なくとも共同で確認できる状態）
- 本番URL（想定）: `https://redish-sfa.vercel.app`

### Supabase

- プロジェクトへのメンバー招待
- テーブル/RLS/認証設定の閲覧・変更権限（権限分離する場合は方針を共有）
- `supabase/migrations` の適用方針（誰がいつ適用するか）

### Google（GCP）

`docs/05_環境構築・デプロイ/setup_guide_supabase_vercel_github.md` の記載を正とする。

- OAuth 2.0 クライアント（Web）作成済み
- 承認済みのJavaScript生成元
  - `http://localhost:3000`
  - `https://redish-sfa.vercel.app`
- 承認済みのリダイレクトURI（末尾まで一致させる）
  - `http://localhost:3000/api/auth/callback/google`
  - `https://redish-sfa.vercel.app/api/auth/callback/google`

### Google Spreadsheet

`docs/01_企画書/SFA_Webアプリ化_企画書・要件定義書.md` のチェックリスト（環境準備）を正とする。

- 既存SFAスプレッドシートへの閲覧/編集権限
- Sheets API 用の認証方式（サービスアカウント等）と、必要な共有設定

## ローカルで必要な設定（最低限）

### 実行起点

- `frontend/` がアプリの起点（Vercelの Root Directory も `frontend`）

### .env.local

`.env.local` は `frontend/` 配下に作成する。

必須の環境変数（詳細は setup guide を正とする）:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `NEXTAUTH_URL`（ローカルは `http://localhost:3000`）
- `NEXTAUTH_SECRET`
- `GOOGLE_CLIENT_ID`
- `GOOGLE_CLIENT_SECRET`
- `ALLOWED_EMAIL_DOMAINS`（例: `redish.jp`）

## 作業の進め方（最小）

- ブランチ戦略: `feature/...` で作業しPR
- 作業の開始/終了時: `PROJECT_CONTROL.md` の履歴を更新（変更ファイルも追記）
- コミット前: `npx tsc --noEmit` で型エラー0を確認（必須）

## 1週目のゴール（推奨）

- ローカル起動、Googleログイン成功
- `calls` / `deals` / `leads` のいずれか1画面で、データ取得までの動線確認
- 次スプリントの範囲合意（担当/期限/DoD）
